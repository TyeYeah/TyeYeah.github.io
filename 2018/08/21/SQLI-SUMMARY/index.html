MYSQL注入引入

手工注入步骤

  • 判断是否存在注入,注入是字符型还是数字型
    在输入处输入 1（测试数字型注入）、1’（测试字符型注入）、1” （测试字符型注入） 再拼接and 1=1或or 1=1等永真式来测试,如果哪种情况下页面出现不同的显示则说明存在相应情况注入

  • 猜解SQL查询语句中的字段数
    需要用到order by来逐个测试字段数,在前一阶段基础上构造如1’ order by 1 #、1’ order by 2 #、 1’ order by 3 #…,常用二分法来猜解,这里的#代表着注释符

  • 确定显示的字段顺序
    类似前一步骤,可构造如 1’ union select 1,2,… #用到的知识是union联合查询需要前后两个查询语句取出的列数相同

  • 获取当前数据库
    输入类似 1’ union select 1,database() # 进行查询

  • 获取数据库中的表
    使用类似 xx’ union select 1,table_name from information_schema.tables where table_schema=’数据库名’– ,此处–也是注释符

  • 获取表中的字段名
    输入类似 xx’ union select 1,column_name from information_schema.columns where table_name=’表名’–

  • 下载数据
    xx’ union select 字段名 from 表名 –

这个变量用于限制数据导入和导出操作造成的影响,例如由 LOAD DATA、SELECT…INTO OUTFILE 语句和 LOAD_FILE() 函数执行的操作。
1、如果变量设置为目录的名称,则服务器会将导入和导出操作限制在跟这个目录中一起使用。这个目录必须存在,服务器不会自己创建它。

3、如果变量设置为 NULL,那么服务器就会禁用导入和导出操作。这个值从 MySQL 5.5.53 版本开始允许。
在 MySQL 5.5.53 之前,此变量默认为空,因此我们就可以使用这些函数。但是在 5.5.53 之后的版本中,NULL 值会禁用这些功能。(根据两台测试机器不同版本的 mysql 来判断,会默认为 mysql 的一个 /upload 根目录下)

宽字节注入

宽字节即多个字节表示的字符
php中就有addslashes等函数来防范sql注入,即在每个单双引号和反斜杠,以及值NULL的前面,加上一个反斜杠,这样可以防止单双引号的提前闭合等
如果mysql用gbk作为编码方式,两个字符算作一个汉字,而前一个字符的ASCII码值要大于128,才到汉字的范围
经常使用%df来进行宽字节注入,如1%df’ and 1=1
这样最终的查询语句会变成select * from xxx where id=’1運’and 1=1
不只是%df,还可以是%a1、%aa等,只要ASCII码大于128
但是如果编码变化,如果为gb2312,则方法也不同,因为gb2312编码中对高低位字节ASCII码值要求不同,而\的ascii码不在低位范围中,所以需要换个方法

而在防止sql注入的时候,一定要注意编码转换的过程中,会出现疏漏导致引号逃逸
可以将addslashes函数替换为mysql_real_escape_string,此时需要将php与mysql连接的编码方式,以及mysql中编码方式统一为gbk才可有效
还可以在所有sql语句前指定一下连接的形式是二进制：
mysql_query(“SET character_set_connection=gbk, character_set_results=gbk,character_set_client=binary”, $conn);
当我们的mysql接受到客户端的数据后,会认为他的编码是character_set_client,然后会将之将换成character_set_connection的编码,然后进入具体表和字段后,再转换成字段对应的编码。
然后,当查询结果产生后,会从表和字段的编码,转换成character_set_results编码,返回给客户端。
所以,我们将character_set_client设置成binary,就不存在宽字节或多字节的问题了,所有数据以二进制的形式传递,就能有效避免宽字符注入。
以上多参考于https://www.cnblogs.com/lcamry/articles/5625276.html

而在某些情况,网站服务端选择使用二次编码,如对url在客户端两次url编码,之后服务端也两次url解码,可是过滤的函数使用次序不当,就会产生疏漏
例：将%2527送至服务端,先进性一次url解码成%27（%25url解码后为%）,这个时候使用过滤函数没有问题,但是这之后再url解码成 ‘ 如果直接带入sql查询语句,不做过滤,就会出现注入漏洞

还有二次注入,指的是服务端虽然对接受的数据进行了转义等的处理,但还是能将构造好的脏数据存放到表内。而服务端通常对数据库中的内容在使用的时候不加过滤和转义,这个时候拼接入sql语句就会产生注入漏洞

多语句,堆叠注入
Select from a where id = 1;select from flag;
Sql server中为Select from a where id = 1 select from flag;
对于php-mysql来说,通过PDO_MYSQL和MYSQLi方式与数据库交互地可以使用堆叠注入,但通过mysql_query进行交互的不可以

根据数字数据判断注入点
若原始值时2,使用3-1或者1+1注入看结果,或者使用67-ASCII（’A’）或51-ASCII（1）,如果相同则存在注入
51-ASCII（1）应用在过滤单引号的情况下,数据库会隐含的将数字数据转化为字符串数据,字符1的ascii值为49,所以 表达式等于2
